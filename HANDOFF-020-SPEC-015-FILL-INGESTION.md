# HANDOFF 020 — 스펙 015 라이브 체결 동기화 출시 (2026-05-27)

PR #73 머지 커밋 `e746f52`. **접수된 라이브 주문이 실제로 체결됐는지를 브로커에서
다시 조회해 장부(FILL 감사·`fills` 테이블·보유 캐시·주문 상태)에 반영하는** 마지막
고리를 채웠습니다. 그동안 라이브 주문은 `SUBMITTED`(브로커 접수)에서 멈추고 실제
체결을 추적하는 코드가 0건이었습니다. 운영자 지시("세계 최고 수준 작업 분석·우선순위
·자율 수행")의 코드 분석 1순위로 선택(운영자 확인). 직전 세션이 같은 지시로 출시한
스펙 014를 라이브에서 **실제로 작동**하게 만드는 직접 후속입니다.

## 왜 1순위였나 (실거래 전 가장 큰 구멍)

`FILL` 감사 이벤트·`fills` 테이블·`positions.update_from_fill`이 **정의·조회만**
되고 라이브 writer 가 0건이라(유일한 writer 는 백테스트 리플레이), 이미 출시된 안전·
측정 기능이 라이브에서 통째로 눈을 뜨지 못했습니다:

- **스펙 014 손실 서킷 브레이커**가 라이브에서 트립 불가 — 스펙 011 실현 손익을
  읽는데 라이브 FILL 이 0건이라 손익이 항상 0으로 보였음.
- **정합성(reconciliation)**이 매 세션 불일치→halt — 보유 캐시가 체결로 갱신 안 됨.
- **스펙 011 성과·스펙 005 튜너**가 라이브에서 신호 0 — 측정 기반 자율 성장(헌법 X)
  의 입력 공백.
- 부분 체결·미체결 LIMIT·접수 후 거부가 전부 비가시.

## 한 줄 요약

- **브로커 체결 조회**: `broker/overseas.get_order_executions`(KIS `inquire-ccnl`,
  읽기 전용 GET) + `BrokerExecution` 모델. 후보 필드명 폴백(실전/모의 키 변형 대응).
- **새 모듈 `execution/fill_sync.py`**:
  - 순수 함수 `plan_fill_ingestion(open_orders, executions, recorded_qty_by_corr)`
    — 누적 체결량 대비 추가분(delta)만큼 FILL + 상태 전이 계획을 결정론적으로 반환.
  - async `sync_fills` — 열린 주문 로드 → 체결 조회 → 계획 적용(FILL 감사 +
    `fills` INSERT OR IGNORE + `positions.update_from_fill` + 상태 전이). 0건이면
    브로커 미호출. 오류는 격리(ERROR 감사, 거래 무중단).
  - **멱등**: `kis_fill_id = "{kis_order_id}:{누적체결량}"` + 이미 기록된 양만큼
    delta=0 → 재폴링 안전(이중 안전).
- **상태 전이**: `SUBMITTED`→`PARTIALLY_FILLED`→`FILLED`, 브로커가 종료(취소/만료)를
  명시적으로 보고하고 미완이면 `EXPIRED` + `CANCEL` 감사. 종료 신호 없으면 무전이(보수적).
- **워커 연결**: `tick`에서 브레이커 점검 이후·룰 평가 전에 **라이브 전용** cadence
  (5초)로 호출. paper 모드는 호출 안 함(동작 무변경). 열린 주문 0건이면 브로커 미호출.
- **CLI**: `auto-invest fills [--sync]` — `--sync`=1회 동기화(--env 필요)+적용 건수,
  무인자=읽기 전용 요약(열린 주문·최근 체결). 종료 코드 0/1/2.
