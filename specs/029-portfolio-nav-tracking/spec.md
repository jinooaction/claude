# 스펙 029 — 라이브 포트폴리오 순자산(NAV) 추적

상태: 슬라이스 1 출시 완료 · 슬라이스 2 진행 중
작성일: 2026-05-30
운영자 지시: "세계 최고 수준이 되기 위한 작업 분석 … 특히 내 포트폴리오와 현재 자산
수준을 기준으로 판단하고 성장시키는 관리 능력이 중요하지 않을까?" + "이어서 바로 진행해"

## 배경 (왜)

이 시스템은 알파(신호)·사이징·팩터·백테스트 통계는 세계 최고 데스크에 근접했다
(스펙 016~028). 그러나 운영자가 짚은 축 — **"내 실제 포트폴리오와 현재 자산 수준을
기준으로 운용·성장 관리"** — 에는 세 가지 구조적 빈칸이 있다.

1. **자본 기준이 고정·낡음**: 워커는 시작 시 KIS 예수금을 한 번 읽어
   `total_capital_usd` 로 두고, 그 뒤로는 **상수**로 쓴다(`worker/loop.py`).
   모든 리스크 캡(K1)은 "자본의 N%"인데(`risk/gates.py:113`), 기준 자본이 안 갱신되니
   자산이 불어도 포지션이 안 커지고(성장 미반영), 자산이 줄어도 캡이 안 줄어든다
   (방어 미작동). 이는 "현재 자산 수준 기준 운용"의 정반대다. → **슬라이스 2가 해결.**

2. **포트폴리오 단일 시점 모델 없음**: 현금·보유평가·미실현손익을 합친 **현재 순자산
   (NAV = Net Asset Value)** 을 하나로 보여주는 권위 있는 모델이 없었다. → **슬라이스 1
   (출시 완료)이 해결.**

3. **장부 vs 브로커 정합성의 자산 관점 부재**. → 슬라이스 1이 드리프트로 해결.

## 슬라이스 구성

- **슬라이스 1 (출시 완료, main `cfd7e3d`)**: 순자산(NAV) 스냅샷 모듈 + `auto-invest
  portfolio` CLI + K4 추가-전용 감사 이벤트. 측정 전용.
- **슬라이스 2 (이 문서, 자산 인식 자본 기준 — 옵트인)**: 워커가 매 사이클 라이브
  순자산으로 게이트에 넘기는 **유효 자본(effective capital)** 을 갱신해 캡이 실제 자산을
  추종. 기본 끔(byte 동일), 하락은 항상 적용(방어), 상승은 옵트인 + 상한 클램프.
- **슬라이스 3 (성장 추적)**: 슬라이스 1 스냅샷 시계열 → 미실현 포함 시가평가 자산곡선.

이 문서는 슬라이스 1·2·3을 누적해 다룬다.

## 슬라이스 3 — 성장 추적 (미실현 포함 시가평가 자산곡선)

### 배경 (왜)

성과 엔진(스펙 011)의 자산곡선은 **실현손익만** 누적한다(`performance/engine.py:412`,
"미실현은 과거 시세 없이 시점별 평가 불가"). 슬라이스 1이 `PORTFOLIO_NAV_SNAPSHOT`
감사 이벤트로 **미실현 포함 순자산을 시점별로** 남기기 시작했으므로, 이제 그 시계열을
이어 붙이면 실현+미실현을 합친 진짜 시가평가(mark-to-market) 자산곡선을 그릴 수 있다.
운영자가 "내 자산이 시간에 따라 실제로 어떻게 성장했나"를 보는 마지막 조각이다.

### 사용자 시나리오

#### US6 — 운영자가 순자산 성장 추세를 한눈에 본다 (P1)

`auto-invest growth --mode live` 를 실행하면, 기록된 NAV 스냅샷들을 시간순으로 이어
시작 순자산·현재 순자산·총수익률·최대낙폭·기간(일수)·연환산수익률(CAGR)을 표로 본다.
스냅샷이 2개 미만이면 "추세 측정 불가"로 우아하게 강등한다.

### 기능 요구사항 (FR)

- **FR-14**: `portfolio/growth.py` 가 audit_log 의 `PORTFOLIO_NAV_SNAPSHOT` 행을 모드별·
  기간별로 읽어 시간순 (시각, 순자산) 점열로 만든다. 읽기 전용(SELECT만).
- **FR-15**: `compute_growth(points)` 순수 함수가 점열에서 시가평가 자산곡선 지표를
  결정론적으로 계산한다 — 시작/현재 순자산, 총수익률%(스펙 008 `total_return_pct`),
  최대낙폭%(`max_drawdown_pct`), 스냅샷 수, 기간(첫→마지막 일수), CAGR(기간 ≥ 1일).
- **FR-16**: 스냅샷 0·1개면 추세 None(측정 불가). 순자산에 0 이하가 섞이면 낙폭/CAGR은
  None 으로 강등(곡선이 양수일 때만 계산, 스펙 008 계약과 동일).
- **FR-17**: `auto-invest growth` CLI — `--mode`(live|paper), `--db`, `--format`
  (text|json), `--since`/`--window`. read-only(`PRAGMA query_only=ON`).

### 성공 기준 (SC)

- **SC-17**: NAV 스냅샷 [$10,000 → $11,000 → $12,100] → 총수익률 +21%, 시작 $10,000,
  현재 $12,100.
- **SC-18**: [$10,000 → $12,000 → $9,000 → $11,000] → 최대낙폭 25%(=12,000→9,000).
- **SC-19**: 스냅샷 1개 → 추세 None(측정 불가), 예외 없음.
- **SC-20**: 같은 입력 → 같은 출력(결정론). 스펙 008 metrics 재사용(단일 잣대).
- **SC-21**: 기간 365일·총수익률 +21% → CAGR ≈ +21%(1년). 기간 0일이면 CAGR None.

### 안전 경계 (슬라이스 3)

- **Kernel 터치 0건.** `portfolio/growth.py` 신규(비커널), `cli.py` 커맨드 추가(비커널).
  감사 스키마 무변경(읽기만 — 새 이벤트 0건). 게이트·사이징·주문 라우터·워커 무변경.
- **읽기 전용.** SELECT만, `PRAGMA query_only=ON`. 주문/halt/자본 갱신 0건.
- **단일 잣대**(헌법 X.2) — 자산곡선 지표는 스펙 008 `backtest/metrics.py` 재사용
  (백테스트·라이브 같은 공식). 결정론·LLM 미사용. dry-run/paper 그대로.

## 슬라이스 2 — 자산 인식 유효 자본

### 사용자 시나리오

#### US4 — 자산이 불면 포지션도 같이 큰다 (성장, 옵트인)

자율 캐너리가 수익을 내 순자산이 시작 자본보다 커지면, 운영자가 옵트인한 경우 캡 기준이
현재 순자산을 따라 올라가 같은 룰이 (상한 안에서) 더 큰 포지션을 잡을 수 있다. 복리 성장의
토대다. 상한(`max_growth_factor`, 기본 2배)으로 폭주를 막는다.

#### US5 — 자산이 줄면 캡도 같이 준다 (방어, 항상)

순자산이 시작 자본보다 작아지면 — 옵트인 여부와 무관하게 — 캡 기준이 현재 순자산으로
낮아져 손실 구간에서 포지션이 자동으로 작아진다. 방어는 협상 대상이 아니므로 항상 켜져
있다.

### 기능 요구사항 (FR)

- **FR-08**: `portfolio/nav.py` 의 순수 함수 `effective_capital(starting, nav, *,
  growth_enabled, max_growth_factor)` 이 (시작 자본, 현재 순자산, 옵트인 플래그, 상한)
  으로 게이트에 넘길 유효 자본을 결정론적으로 계산한다.
- **FR-09**: 하락(nav < starting)은 **항상** 유효 자본 = nav (방어). `growth_enabled`
  와 무관.
- **FR-10**: 상승(nav > starting)은 `growth_enabled=True` 일 때만 반영하며, 유효 자본은
  `min(nav, starting × max_growth_factor)` 로 클램프. `growth_enabled=False` 면 유효
  자본 = starting (시작 자본이 천장, 슬라이스 1 이전과 동일).
- **FR-11**: nav 가 None/0 이하(조회 실패·미측정)면 유효 자본 = starting (폴백, 거래
  무중단).
- **FR-12**: 워커가 매 사이클(캐던스로 묶음) 라이브 순자산을 조회해 유효 자본을 캐시하고,
  게이트 호출 시 `total_capital_usd` 대신 이 값을 넘긴다. paper 모드는 비활성(가상 계좌
  — 항상 시작 자본). 옵트인 끄면(`capital_tracking_enabled=False`) 기존 동작 byte 동일.
- **FR-13**: 자본 갱신 시 추가-전용 감사 이벤트(`EFFECTIVE_CAPITAL_UPDATED`)를 남겨
  포렌식·자율 튜너가 소비. 값이 안 바뀌면 기록 안 함(노이즈 방지).

### 성공 기준 (SC)

- **SC-09**: 시작 $10,000, NAV $8,000 → 유효 자본 $8,000 (방어, 옵트인 무관).
- **SC-10**: 시작 $10,000, NAV $15,000, growth 끔 → 유효 자본 $10,000 (시작이 천장).
- **SC-11**: 시작 $10,000, NAV $15,000, growth 켬, 상한 2배 → 유효 자본 $15,000.
- **SC-12**: 시작 $10,000, NAV $25,000, growth 켬, 상한 2배 → 유효 자본 $20,000 (클램프).
- **SC-13**: NAV None → 유효 자본 = 시작 자본 (폴백).
- **SC-14**: 유효 자본이 게이트 캡 계산에 실제로 흘러간다 — NAV $8,000·per_trade 5% →
  per-trade 캡 $400(=$8,000×5%), 시작 $10,000 기준 $500이 아님.
- **SC-15**: `capital_tracking_enabled=False`(기본) → 워커는 NAV 조회 안 하고 시작 자본
  그대로 — 기존 워커 테스트 byte 동일(회귀 무손상).
- **SC-16**: paper 모드 → 자본 추적 비활성(가상 계좌라 라이브 NAV 무의미).

### 안전 경계 (비협상)

- **Kernel 터치 0건.** `portfolio/nav.py` 에 순수 함수 추가(비커널), `worker/loop.py`
  (비커널 — 커널은 `worker/schedule.py`)에서 게이트에 넘기는 자본값만 교체, `audit.py`
  추가-전용 이벤트(K4). **게이트(`risk/gates.py`)·캡 스키마(`config/caps.py`)·사이징·
  주문 라우터 무변경** — 캡 계산식·퍼센트는 그대로다.
- **K1이 여전히 천장.** 유효 자본은 게이트에 넘기는 "자본 기준"일 뿐, 게이트가 거부하는
  로직은 무변경. 상승 반영은 `max_growth_factor` 로 하드 클램프돼 폭주 불가.
- **방어는 항상, 성장은 옵트인.** 하락은 항상 캡을 줄이고, 상승은 명시적 옵트인 + 상한.
- **기본 끔 → byte 동일.** `capital_tracking_enabled` 기본 False. 켜지 않으면 워커는
  NAV 를 조회조차 안 하고 시작 자본을 그대로 쓴다(회귀 무손상).
- **자율 전환 아님.** 이건 라이브 모드를 켜거나 자본을 새로 넣는 게 아니라, 이미 라이브인
  워커가 "캡 기준을 실제 자산에 맞춘다"는 것. AUTO_INVEST_MODE 토글 무관.
- **추가-전용 감사**(헌법 IV), 결정론, LLM 미사용, dry-run/paper 그대로.
