# 스펙 029 — 라이브 포트폴리오 순자산(NAV) 추적

상태: 진행 중 (슬라이스 1 — 측정 전용)
작성일: 2026-05-30
운영자 지시: "세계 최고 수준이 되기 위한 작업 분석 … 특히 내 포트폴리오와 현재 자산
수준을 기준으로 판단하고 성장시키는 관리 능력이 중요하지 않을까?"

## 배경 (왜)

이 시스템은 알파(신호)·사이징·팩터·백테스트 통계는 세계 최고 데스크에 근접했다
(스펙 016~028). 그러나 운영자가 짚은 축 — **"내 실제 포트폴리오와 현재 자산 수준을
기준으로 운용·성장 관리"** — 에는 세 가지 구조적 빈칸이 있다.

1. **자본 기준이 고정·낡음**: 워커는 시작 시 KIS 예수금을 한 번 읽어
   `total_capital_usd` 로 두고, 그 뒤로는 **상수**로 쓴다(`worker/loop.py:370`).
   모든 리스크 캡(K1)은 "자본의 N%"인데(`risk/gates.py:113`), 기준 자본이 안 갱신되니
   자산이 불어도 포지션이 안 커지고(성장 미반영), 자산이 줄어도 캡이 안 줄어든다
   (방어 미작동). 이는 "현재 자산 수준 기준 운용"의 정반대다.

2. **포트폴리오 단일 시점 모델 없음**: 현금·보유평가·미실현손익을 합친 **현재 순자산
   (NAV = Net Asset Value)** 을 하나로 보여주는 권위 있는 모델이 없다. 성과 엔진
   (스펙 011)은 종목·룰별로 쪼개져 있고, 자산곡선은 **실현손익만** 누적한다
   (`performance/engine.py:412`). "내 자산이 지금 얼마인지"의 단일 진실이 없다.

3. **장부 vs 브로커 정합성의 자산 관점 부재**: 정합성 검증(스펙 001)은 포지션 수량
   불일치를 보지만, **순자산(현금+평가) 관점의 드리프트**(입출금·배당·수수료·환차로
   생긴 차이)는 측정하지 않는다.

세계 최고 수준 데스크와의 차이는 더 화려한 알파가 아니라, **"낡은 고정 숫자가 아니라
살아있는 실제 순자산을 기준으로 사이징·리스크·성장을 관리한다"** 는 점이다. 개선의
전제는 측정이다(스펙 028과 같은 규율). 그래서 이 스펙은 **측정부터** 한다.

## 슬라이스 구성

- **슬라이스 1 (이 문서, 측정 전용)**: 순자산(NAV) 스냅샷 모듈 + `auto-invest
  portfolio` CLI + K4 추가-전용 감사 이벤트. **주문 경로·게이트·사이징 한 바이트도
  안 건드린다.** 브로커 잔고(권위 있는 실제값)와 내부 장부(audit_log 재구성)를 함께
  담고, 둘의 드리프트를 보고한다.
- **슬라이스 2 (후속, 자산 인식 자본 기준 — 옵트인)**: 워커가 매 사이클 라이브
  순자산으로 `total_capital_usd` 를 갱신해 캡이 실제 자산을 추종. 기본 끔(byte 동일),
  하락은 항상 적용(방어), 상승은 옵트인.
- **슬라이스 3 (후속, 성장 추적)**: 슬라이스 1 스냅샷을 시계열로 모아 **미실현 포함
  시가평가 자산곡선** → 진짜 순자산 성장 추세.

이 문서는 슬라이스 1만 다룬다.

## 사용자 시나리오

### US1 — 운영자가 "내 자산 지금 얼마"를 한눈에 본다 (P1)

운영자가 `auto-invest portfolio --env .env` 를 실행하면, **현금 + 종목별 평가금액 +
총 순자산(NAV)** 이 표로 출력된다. 종목마다 수량·평균단가·현재가·평가금액·비중(%)이
보인다. 시세 조회가 실패하면 우아하게 강등(현금·장부만 표시)된다.

### US2 — 장부와 브로커의 순자산 드리프트를 본다 (P1)

내부 장부(체결로 재구성한 보유)와 브로커 실제 보유가 다르면(입출금·배당·수수료·환차
미반영, 또는 누락 체결) 그 차이를 **순자산 관점**으로 보여준다. 큰 드리프트는 운영자가
조사해야 할 신호다(이 슬라이스는 보고만 — halt/주문 0건).

### US3 — 자율 튜너·후속 슬라이스가 소비할 NAV 스냅샷을 남긴다 (P2)

`--snapshot` 지정 시 결과를 추가-전용 `PORTFOLIO_NAV_SNAPSHOT` 감사 이벤트로 1건
기록한다. 슬라이스 3(성장 추적)이 이 시계열을 읽어 미실현 포함 자산곡선을 그린다.
기본은 미기록(순수 계산).

## 기능 요구사항 (FR)

- **FR-01**: `portfolio/nav.py` 의 순수 함수가 (브로커 잔고, 브로커 보유, 내부 장부
  포지션, 현재 시세 dict) 를 입력받아 `NavSnapshot` 을 결정론적으로 계산한다. 외부
  API 호출 없음 — CLI 계층이 KIS 를 조회해 주입한다(테스트 가능성, 스펙 011 패턴).
- **FR-02**: `NavSnapshot` 은 ① 현금(USD), ② 종목별 보유(수량·평균단가·현재가·평가
  금액·비중%·미실현손익), ③ 총 평가금액(보유 합), ④ 총 순자산(NAV=현금+평가),
  ⑤ 권위 출처 표시(브로커 우선, 폴백 장부)를 담는다.
- **FR-03**: 종목 비중(%)은 평가금액 / 총 순자산 × 100, Decimal 결정론. 총 순자산이
  0 이면 비중은 None(0 나눗셈 방지).
- **FR-04**: 드리프트 — 같은 종목에 대해 (브로커 수량 − 장부 수량), (브로커 평가 −
  장부 평가) 를 종목별·합계로 계산한다. 한쪽에만 있는 종목도 드러낸다.
- **FR-05**: 시세 조회 실패/부분 실패 시 우아한 강등 — 마크 없는 종목은 미실현 None,
  평가금액은 평균단가 대용(보수적 표기)이며 `unmarked_symbols` 로 분리한다.
- **FR-06**: `auto-invest portfolio` CLI — `--env`(KIS 조회), `--db`, `--format`
  (text|json), `--no-marks`, `--snapshot`, `--mode`(live|paper). read-only:
  `PRAGMA query_only=ON` 로 DB 쓰기 차단(`--snapshot` 제외).
- **FR-07**: `--snapshot` 지정 시에만 `PORTFOLIO_NAV_SNAPSHOT` 추가-전용 이벤트 1건
  기록. 기존 이벤트/row 무변경(K4 추가-전용, 헌법 IV).

## 성공 기준 (SC)

- **SC-01**: 현금 $1,000 + AAPL 10주(평단 $100, 현재 $120) → 평가 $1,200, NAV
  $2,200, AAPL 비중 ≈ 54.5%.
- **SC-02**: 총 순자산 0 → 모든 비중 None, 0 나눗셈 예외 없음.
- **SC-03**: 시세 일부 누락 → 해당 종목 미실현 None·`unmarked_symbols` 에 포함,
  나머지 종목은 정상 평가. NAV 는 누락 종목을 평균단가로 보수 평가.
- **SC-04**: 브로커 보유 AAPL 10 vs 장부 8 → 종목 드리프트 +2, 드리프트 목록에 등장.
- **SC-05**: 브로커에만 있는 TSLA(장부 0) → 드리프트에 broker_only 로 표기.
- **SC-06**: `--snapshot` → `PORTFOLIO_NAV_SNAPSHOT` 정확히 1건, 기존 이벤트 카운트
  불변. `--snapshot` 없으면 0건.
- **SC-07**: 같은 입력 → 같은 출력(결정론). Decimal 경로, 부동소수 없음.
- **SC-08**: 권위 출처 — 브로커 잔고가 있으면 NAV·현금은 브로커 값, 없으면(오프라인/
  실패) 장부 + 마크로 폴백하고 `source="ledger"` 로 표기.

## 안전 경계 (비협상)

- **Kernel 터치 0건.** `portfolio/nav.py` 신규(비커널), `cli.py` 커맨드 추가(비커널),
  `audit.py` 추가-전용 이벤트(K4). 게이트(`risk/gates.py`)·사이징(`strategy/`)·
  주문 라우터(`execution/`)·워커 루프(`worker/loop.py`) **무변경**.
- **측정 전용.** 주문을 새로 내거나 취소·정정하지 않는다. halt 세팅 안 함. 자본
  기준(`total_capital_usd`) 갱신 안 함(그건 슬라이스 2). 드리프트가 커도 보고만 한다.
- **읽기 전용 DB** (`--snapshot` 의 단일 추가 row 제외). `PRAGMA query_only=ON`.
- **추가-전용 감사** (헌법 IV) — 새 이벤트 타입 1개, 기존 row/타입 무변경. DB
  마이그레이션 불필요(audit_log 는 payload_json 자유 스키마).
- **결정론·LLM 미사용.** 라이브 전환 토글 무관. dry-run 그대로.
- **단일 잣대** (헌법 X.2) — 평균단가 원가 규약은 스펙 009/011 과 동일(`PositionState`
  재사용), 미실현 = (현재가 − 평단) × 수량.
