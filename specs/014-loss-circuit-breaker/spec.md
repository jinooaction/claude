# Feature Specification: Live Loss Circuit Breaker (자동 손실 차단)

**Feature Branch**: `claude/adoring-edison-AIkbz` (spec dir `014-loss-circuit-breaker`)
**Created**: 2026-05-26
**Status**: Draft
**Input**: 운영자 지시 — "세계 최고 수준이 되기 위한 작업 분석하고 우선순위에 따라 자율 수행해." 분석 1순위로 **라이브 손실 서킷 브레이커**를 선택(운영자 확인).

## 배경 (왜 이 기능인가)

오늘 시스템의 위험 통제는 전부 **노출(exposure) 상한**이다: 거래당·종목당·전체
포지션 캡(`risk/gates.py`, 헌법 원칙 I)과 운영자 수동 halt 플래그뿐이다. **손실
크기에 반응하는 자동 차단이 0건**이다 — 포지션 캡 안에서도 시장이 무너지면 손실은
얼마든지 누적될 수 있고, 그걸 멈추는 유일한 장치는 운영자가 손으로 `halt`를 거는
것뿐이다.

세계 최고 수준의 자율 거래 시스템은 **손실 기반 서킷 브레이커**를 반드시 가진다:
하루 손실이 한도를 넘거나 자산이 시작 자본 대비 일정 비율 이상 무너지면 **사람
개입 없이 즉시 새 주문을 멈춘다.** 이 공백이 실거래 전환 전 안전 기반의 가장 큰
구멍이다.

이 기능은 그 장치를 만든다: 기존 측정(스펙 011 성과 엔진)과 기존 정지 장치(halt
플래그)를 잇는 **자동 손실 차단 루프**를 워커 안에 넣는다. 손실이 한도를 넘으면
워커가 스스로 halt 플래그를 세우고 추가-전용 감사 이벤트를 남긴 뒤 새 주문을
멈춘다.

**핵심 안전 불변**:
- **순수 방어적** — 브레이커는 **거래를 멈추기만** 한다. 노출을 늘리거나 주문을
  내거나 포지션을 청산하지 않는다. 트립의 유일한 효과는 기존 halt 플래그를 세우는
  것(= 새 주문 차단)이다. 블래스트 반경은 "거래 중단" 한 방향뿐이다.
- **한도는 Kernel 보호(K1)** — 손실 한도는 `config/caps.py`(K1 포지션 사이징)에
  둔다. 따라서 자율 튜너(스펙 005)는 이 한도를 **절대 자동으로 완화할 수 없다**
  (Kernel 교집합 = L4 인간 머지 강제). 세계 최고 수준 안전 통제가 가져야 할 성질.
- **측정은 스펙 011 정의 재사용** — 실현/미실현 손익은 성과 엔진(`performance/
  engine.py`)의 같은 정의(평균단가·문자열 Decimal)로 계산한다(헌법 원칙 X: 한
  잣대). 백테스트·캐너리·라이브·성과 리포트와 일관.
- **승격 0건** — 브레이커는 검증·승격·배포와 무관하다. 자동 머지/캐너리/실거래
  토글을 건드리지 않는다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 일일 손실 한도 초과 시 자동 정지 (Priority: P1)

워커가 거래 중 오늘 실현 손실이 시작 자본의 한도 비율(`daily_loss_limit_pct`)을
넘어선다. 다음 틱에서 워커는 손실을 감지해 **스스로 halt 플래그를 세우고**
`CIRCUIT_BREAKER_TRIPPED` 감사 이벤트를 남긴 뒤 새 주문을 멈춘다.

**Acceptance**:
- 오늘(UTC 기준 날짜) 청산으로 실현된 손익 합이 `-(daily_loss_limit_pct% ×
  시작 자본)` 이하이면 브레이커가 트립한다.
- 트립 시: (1) halt 플래그가 세워지고(사유에 한도·실측값 포함), (2)
  `CIRCUIT_BREAKER_TRIPPED` 감사 row가 정확히 한 번 기록되며, (3) 그 틱은 새
  주문을 평가하지 않고 `skipped_reason="circuit_breaker_tripped"`로 끝난다.
- 한도 미만이면 트립하지 않고 정상 거래한다.
- "오늘 실현 손실"은 스펙 011 성과 엔진의 `realized_trades`에서 청산일이 오늘인
  거래의 손익 합으로 계산한다(시작 자본 이전 매수의 원가 기준도 정확히 반영).

### User Story 2 — 전체 자산 낙폭 한도 초과 시 자동 정지 (Priority: P1)

워커의 현재 자산(시작 자본 + 누적 실현 손익 + 현재 미실현 손익)이 시작 자본 대비
`max_total_drawdown_pct%` 이상 무너진다. 워커는 다음 틱에서 이를 감지해 자동 정지한다.

**Acceptance**:
- `현재 자산 ≤ 시작 자본 × (1 − max_total_drawdown_pct/100)`이면 트립한다.
- 미실현 손익은 보유 종목별 **최근 저장 바의 종가**를 시세로 사용해 계산한다.
  시세가 없는 종목은 미실현 0으로 보수 처리하고(과대 손실로 오트립하지 않음)
  사유 메타데이터에 표시한다.
- 두 한도(일일·낙폭) 중 하나라도 걸리면 트립하며, 사유에 어떤 한도가 걸렸는지
  명시한다. 둘 다 걸리면 둘 다 사유에 남긴다.

### User Story 3 — 트립 상태의 멱등성과 운영자 복구 (Priority: P2)

브레이커가 이미 트립해 halt가 걸린 뒤 워커가 계속 돌아도 **중복 트립/중복 감사
row를 만들지 않는다.** 운영자는 기존 `auto-invest resume` 명령으로 halt를 해제해
거래를 재개한다(브레이커는 별도 복구 명령을 추가하지 않는다).

**Acceptance**:
- halt가 이미 걸려 있으면 틱은 기존대로 `halt_flag_set`로 먼저 끝나므로 브레이커
  평가 자체가 일어나지 않는다(중복 `CIRCUIT_BREAKER_TRIPPED` 0건).
- `resume`로 halt 해제 후, 손실이 여전히 한도 초과면 다음 평가에서 다시 트립할 수
  있다(브레이커는 상태를 기억하지 않는 순수 평가 — 손실이 회복돼야 트립 안 함).

### User Story 4 — 브레이커 상태를 헬스 롤업에서 본다 (Priority: P3)

운영자가 `auto-invest health`를 돌리면 브레이커의 현재 평가(무장/트립, 각 한도까지의
여유)를 **읽기 전용**으로 본다.

**Acceptance**:
- `health`에 브레이커 점검이 추가되어, halt가 브레이커 트립으로 걸린 경우 그 사실과
  사유를 보여준다.
- 이 표면은 100% 읽기 전용 — 평가만 하고 halt/감사/상태를 바꾸지 않는다.

### Edge Cases

- **거래 0건 / 청산 0건**: 손익 0 → 트립 안 함. 신규/조용한 워커에 영향 없음.
- **브레이커 비활성(`circuit_breaker_enabled=false`)**: 평가를 건너뛰고 v1 동작
  그대로(트립 0건). 단 기본값은 활성.
- **시세 전부 결측**: 미실현 0으로 계산 → 실현 손실만으로 판단(보수적, 오트립 안 함).
- **paper 모드 vs live 모드**: 워커 모드에 맞는 체결 이벤트(`ORDER_PAPER_FILLED`
  vs `FILL`)를 읽는다(스펙 011 엔진 규약 그대로).
- **시작 자본 0 이하**: 낙폭 비율 계산 불가 → 낙폭 점검 비활성(일일 한도만), 사유에 표시.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 `config/caps.py`의 `SizingCaps`에 손실 서킷 브레이커
  설정을 **옵션 필드(기본값 포함)**로 추가한다 — 기존 설정 파일을 깨지 않는다
  (추가-전용, 하위 호환): `circuit_breaker_enabled`(기본 true),
  `daily_loss_limit_pct`(기본 10), `max_total_drawdown_pct`(기본 20).
- **FR-002**: 각 손실 한도 비율은 `(0, 100]` 범위여야 하며 위반 시 설정 로딩이
  거부된다(`pydantic` 검증).
- **FR-003**: 시스템은 순수 결정론적 평가 함수를 제공한다 — 시작 자본·오늘 실현
  손익·현재 자산·한도를 입력받아 트립 여부·걸린 한도·실측값을 담은 결정을 반환한다.
  외부 호출·DB 쓰기 없음.
- **FR-004**: 워커는 매 틱(또는 평가 간격마다) halt·세션 점검 **이후** 브레이커를
  평가한다. 트립이면 `set_halt`로 halt 플래그를 세우고 `CIRCUIT_BREAKER_TRIPPED`
  감사 row를 남긴 뒤 그 틱을 새 주문 없이 끝낸다.
- **FR-005**: 브레이커가 읽는 손익은 스펙 011 성과 엔진 함수(`read_fills`,
  `realized_trades`, `compute_performance`)로 계산해 한 잣대(헌법 X)를 지킨다.
- **FR-006**: `CIRCUIT_BREAKER_TRIPPED`는 `persistence/audit.py`(K4)에 **추가-전용**
  으로 더한다 — 기존 이벤트/row를 수정하지 않는다(append-only 불변, 헌법 IV).
- **FR-007**: 브레이커는 노출을 늘리거나 주문/청산을 하지 않는다 — 유일한 부수효과는
  halt 플래그 세팅 + 감사 append.
- **FR-008**: halt가 이미 걸려 있으면 브레이커는 평가되지 않는다(멱등 — 중복 트립
  방지).
- **FR-009 (P3)**: `auto-invest health`는 브레이커 점검을 읽기 전용으로 추가한다.

### Key Entities

- **BreakerLimits**: `enabled`, `daily_loss_limit_pct`, `max_total_drawdown_pct`
  (caps에서 파생).
- **BreakerDecision**: `tripped`(bool), `breached`(걸린 한도 id 목록),
  `reason`(사람용 한 줄), `metadata`(실측값: 오늘 실현 손익·현재 자산·한도 금액 등).
- **CircuitBreakerTrippedPayload**: 감사 페이로드 — 트립 시각·모드·시작 자본·오늘
  실현 손익·현재 자산·걸린 한도·각 한도 금액.

## Success Criteria *(mandatory)*

- **SC-001**: 오늘 실현 손실이 한도 초과면 다음 평가에서 100% 트립하고 halt가
  걸린다(단위·통합 테스트로 검증).
- **SC-002**: 현재 자산이 낙폭 한도 초과면 100% 트립한다.
- **SC-003**: 한도 미만/거래 없음/비활성에서는 트립 0건 — 정상 거래가 막히지 않는다.
- **SC-004**: 트립당 `CIRCUIT_BREAKER_TRIPPED` 감사 row가 정확히 한 번 기록되고,
  halt가 이미 걸린 상태에서는 추가 기록이 0건이다(멱등).
- **SC-005**: 같은 입력 → 같은 결정(순수·결정론적). 브레이커 평가는 DB에
  INSERT/UPDATE/DELETE를 하지 않는다(halt/감사 부수효과는 워커가 트립 시에만 수행).
- **SC-006**: Kernel 터치는 K1(`config/caps.py`, 추가-전용 옵션 필드)과
  K4(`persistence/audit.py`, 추가-전용 이벤트)뿐 — K2·K3·K5·K6·K-meta 터치 0건.

## Constitution Check

- **원칙 I (사이징)**: 강화. 노출 캡에 더해 손실 캡을 추가 — 안전 경계를 넓힌다.
- **원칙 IV (추가-전용 감사)**: 준수. 새 이벤트는 추가-전용.
- **원칙 X (측정 기반)**: 준수. 손익은 스펙 011 엔진의 한 잣대로 계산.
- **원칙 IX (자기수정 경계)**: 강화. 한도가 K1(Kernel)에 있어 튜너가 자동 완화 불가.
- **K-meta**: 미터치(헌법·kernel.toml 변경 없음).
- **실거래/배포 경계**: 미터치. 브레이커는 정지만 하며 승격/배포/실거래 토글과 무관.
