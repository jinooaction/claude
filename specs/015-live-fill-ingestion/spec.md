# Feature Specification: Live Fill Ingestion (라이브 체결 동기화)

**Feature Branch**: `claude/fervent-hamilton-oISbf` (spec dir `015-live-fill-ingestion`)
**Created**: 2026-05-27
**Status**: Draft
**Input**: 운영자 지시 — "세계 최고 수준이 되기 위해서 필요한 작업 확인하고 우선순위 판단해서 자율 수행해." 코드 직접 분석 1순위로 **라이브 체결 동기화**를 선택(운영자 확인).

## 배경 (왜 이 기능인가)

오늘 라이브 주문 경로는 **브로커가 주문을 접수한 시점(`SUBMITTED`)에서 멈춘다.**
주문을 보낸 뒤 그 주문이 **실제로 체결됐는지(전량/부분), 미체결로 남았는지,
거부됐는지**를 다시 조회해 시스템 상태에 반영하는 코드가 없다. 구체적으로:

- 감사 이벤트 `FILL`(`FillPayload`)과 `fills` 테이블은 **정의되고 읽히기만** 한다.
  실제로 `FILL`을 만들어 쓰는 곳은 백테스트 리플레이(`backtest/replay.py`) 하나뿐이고
  **라이브 경로에는 체결 기록자(writer)가 0건**이다.
- KIS 체결 조회 함수(`inquire-ccnl`)가 `broker/overseas.py`에 **아예 없다** —
  `place_order`/`cancel_order`/`get_positions`/`get_balance`만 있다.
- `persistence/positions.py`의 `update_from_fill`·`rebuild_from_fills`는
  **호출자가 0개** — 라이브에서 보유 캐시(`current_positions`)가 체결로 갱신되는
  경로가 없다.

이 공백 때문에 **이미 출시된 안전·측정 기능이 라이브에서 통째로 눈을 뜨지 못한다:**

1. **스펙 014 손실 서킷 브레이커가 라이브에서 작동 불가.** 브레이커는 스펙 011
   성과 엔진의 실현 손익을 읽는데, 라이브 `FILL`이 0건이라 손익이 항상 0으로
   보인다. 시장이 무너져도 트립하지 않는다.
2. **정합성(reconciliation)이 매 세션 불일치→halt.** 보유 캐시가 체결로 갱신되지
   않아 브로커(산 주식 보유)와 로컬(0)이 영구히 어긋난다.
3. **스펙 011 성과 측정·스펙 005 튜너가 라이브에서 신호 0.** 측정 기반 자율
   성장(헌법 원칙 X)의 입력이 라이브에서 비어 있다.
4. **부분 체결·미체결 LIMIT·접수 후 거부가 전부 보이지 않는다.**

세계 최고 수준의 자율 거래 시스템은 **주문 생명주기 전체**(접수→체결/부분체결/
미체결/취소)를 추적한다. 이 기능은 그 마지막 고리를 채운다: 브로커 체결 조회로
실제 체결을 읽어 **멱등하게** `FILL` 감사 이벤트 + `fills` 테이블에 기록하고,
보유 캐시를 갱신하며, 주문 상태를 전이시킨다.

**핵심 안전 불변**:

- **순수 읽기-기반 적재(write only what the broker confirms)** — 이 기능은
  주문을 **내거나 취소하지 않는다.** 브로커가 확인해 준 체결 사실만 로컬에 적는다.
  블래스트 반경은 "이미 일어난 일을 장부에 정확히 기록" 한 방향뿐이다.
- **멱등** — 같은 체결을 두 번 읽어도 `fills` 한 줄만 남는다(`kis_fill_id` UNIQUE +
  누적 체결량 기준 결정론적 키). 폴링을 여러 번 돌려도 안전.
- **Kernel 터치 0건** — 기존 `FILL`/`CANCEL` 감사 이벤트를 재사용(`audit.py` =
  K4 미터치). 새 주문 상태(`FILLED`/`PARTIALLY_FILLED`/`EXPIRED`)는 `orders.state`
  자유 TEXT라 마이그레이션 불필요(K4 미터치). 손댄 파일 전부 비커널.
- **거래를 절대 막지 않음** — 체결 동기화 오류(브로커 호출 실패 등)는 격리되어
  로그만 남기고 워커 거래 루프를 멈추지 않는다(정합성의 INCONCLUSIVE와 같은 철학).
- **페이퍼 동작 무변경** — paper 모드는 `orders` 테이블에 row를 만들지 않고
  `ORDER_PAPER_FILLED`로 자체 체결을 적으므로, 체결 동기화는 **라이브 전용**이고
  paper 경로는 한 줄도 바뀌지 않는다.

## User Scenarios & Testing *(mandatory)*

### US1 — 전량 체결이 장부에 정확히 기록된다 (P1, MVP)

라이브 워커가 100주 매수 주문을 접수(`SUBMITTED`)한 뒤 거래소에서 100주가 전량
체결됐다. 다음 체결 동기화에서 시스템은 브로커 체결 조회로 "100주 @ 체결가"를
읽어 `FILL` 감사 이벤트 + `fills` row를 한 건 남기고, 보유 캐시를 +100주로
갱신하며, 주문 상태를 `SUBMITTED`→`FILLED`로 전이한다.

- **AC1**: 체결 후 동기화 1회 → `fills`에 qty=100 한 줄, `FILL` 감사 이벤트 한 건
  (correlation_id·symbol·rule_id이 원주문과 일치).
- **AC2**: `current_positions`에 해당 종목 100주, 평균단가=체결가.
- **AC3**: 주문 상태 `FILLED`, `order_state_history`에 `SUBMITTED`→`FILLED` 전이.
- **AC4**: 같은 체결 상태로 동기화를 **다시** 돌려도 `fills`는 여전히 한 줄(멱등).

### US2 — 부분 체결이 정확히 누적된다 (P1, MVP)

100주 주문 중 거래소에서 먼저 40주만 체결됐다. 동기화는 40주 `FILL`을 적고 상태를
`PARTIALLY_FILLED`로 둔다. 다음 동기화 때 누적 70주로 늘면 **추가분 30주만** 새
`FILL`로 적는다(이미 적은 40주를 다시 적지 않는다). 최종 100주가 되면 `FILLED`.

- **AC1**: 누적 40 → `fills` 한 줄(40), 상태 `PARTIALLY_FILLED`.
- **AC2**: 누적 70 → `fills` 새 줄(30), 합계 70, 상태 `PARTIALLY_FILLED`.
- **AC3**: 누적 100 → `fills` 새 줄(30), 합계 100, 상태 `FILLED`.
- **AC4**: 보유 캐시는 각 단계에서 누적 체결량과 일치(40→70→100).
- **AC5**: 부분 체결 단계에서 동기화를 반복 호출해도 중복 `FILL` 없음(멱등).

### US3 — 체결 동기화가 스펙 014 서킷 브레이커를 라이브에서 켠다 (P1)

라이브에서 매수→매도로 실현 손실이 발생했다. 체결 동기화가 그 매도 체결을 `FILL`로
기록하면, 스펙 011 성과 엔진이 그 손실을 읽고 스펙 014 브레이커가 한도 초과 시
트립한다(= 이 기능 없이는 트립 자체가 불가능했음).

- **AC1**: 라이브 매도 체결 기록 후 `build_performance_report(mode="live")`가
  실현 손익을 음수로 반영.
- **AC2**: 손실이 일일 한도를 넘으면 워커 틱에서 브레이커가 트립(스펙 014 경로).

### US4 — 미체결/취소가 종료 상태로 전이된다 (P2)

LIMIT 주문이 지정가에 닿지 못해 거래소에서 만료/취소됐고 체결량은 0(또는 일부)이다.
브로커가 그 주문을 "더 이상 열려 있지 않음 + 잔여 미체결"로 보고하면, 동기화는
체결분만큼 기록한 뒤 주문을 `EXPIRED`로 전이하고 `CANCEL` 감사 이벤트를 남긴다.
(보수적: 브로커가 종료를 **명시적으로** 알려줄 때만 전이. 모르면 `SUBMITTED` 유지.)

- **AC1**: 브로커가 잔여 미체결+종료를 보고 → 상태 `EXPIRED`, `CANCEL` 감사 한 건.
- **AC2**: 종료 신호가 없으면 상태는 그대로(추측으로 종료 전이 안 함).
- **AC3**: 부분 체결 후 만료면 체결분 `FILL` + 잔여는 `EXPIRED`(체결 손실 없음).

### US5 — 운영자가 한 줄로 체결을 동기화·조회한다 (P3)

운영자(또는 크론/테스트)가 `auto-invest fills --sync`로 즉시 열린 주문의 체결을
당겨오고, `auto-invest fills`로 최근 열린 주문/체결 현황을 본다.

- **AC1**: `auto-invest fills --sync`가 한 번 동기화하고 적용 건수를 출력, 종료
  코드 0.
- **AC2**: `auto-invest fills`(조회)는 읽기 전용으로 열린 주문·최근 체결 요약 출력.

### Edge Cases

- 브로커 체결 조회 실패 → ERROR 감사 한 건, 워커 계속 진행(거래 무중단), 상태 무변경.
- 브로커가 보고한 누적 체결량 < 이미 기록된 양(되돌림/이상치) → 음수 delta는
  기록하지 않고 데이터 품질 경고만(보유 캐시 훼손 방지).
- `kis_order_id`가 없는(접수 실패) 주문은 동기화 대상에서 제외.
- 체결가가 비양수/파싱 불가 → 그 주문은 건너뛰고 ERROR 감사, 다른 주문은 정상 처리.
- 같은 틱에 열린 주문 0건 → 브로커 호출 없이 즉시 no-op(불필요한 API 절약).

## Requirements *(mandatory)*

### Functional

- **FR-001**: 브로커 계층에 **체결 조회** 함수(`get_order_executions`,
  KIS `inquire-ccnl`)를 추가한다. 계정의 최근 주문별 누적 체결량·평균 체결가
  (+가능하면 매도매수구분·잔여 미체결·종료 여부)를 정규화된 `BrokerExecution`
  목록으로 반환한다. 읽기 전용(GET).
- **FR-002**: 체결 동기화는 로컬 `orders`에서 **열린 주문**(state ∈
  {`SUBMITTED`,`PARTIALLY_FILLED`} 그리고 `kis_order_id` 존재)만 대상으로 한다.
- **FR-003**: 각 열린 주문에 대해 브로커 누적 체결량과 **이미 `fills`에 기록된 양**
  의 차이(delta)가 양수면, delta 만큼의 `FILL`을 **한 건** 기록한다. `kis_fill_id`는
  `"{kis_order_id}:{누적체결량}"`로 결정론적이라 같은 누적 상태 재폴링은 no-op이다.
- **FR-004**: `FILL` 기록은 (1) `FILL` 감사 이벤트 append(correlation_id=원주문,
  symbol·rule_id 일치) + (2) `fills` 테이블 INSERT(append-only, UNIQUE 충돌 무시) +
  (3) `positions.update_from_fill`로 보유 캐시 갱신 — 세 가지를 한 트랜잭션 단위로
  수행한다.
- **FR-005**: 누적 체결량 ≥ 주문 수량이면 주문 상태를 `FILLED`로, 0<체결<주문이면
  `PARTIALLY_FILLED`로 전이하고 `order_state_history`에 기록한다.
- **FR-006**: 브로커가 주문을 **명시적으로 종료**(취소/만료)로 보고하고 체결이
  주문 수량 미만이면, 잔여를 채우지 않고 상태를 `EXPIRED`로 전이 + `CANCEL` 감사
  한 건. 종료 신호가 없으면 전이하지 않는다(보수적).
- **FR-007**: 체결 동기화 핵심 계획 로직은 **순수 함수**로 분리한다 — 입력(열린
  주문, 브로커 체결, 기 기록량) → 출력(적용할 `FILL`·상태 전이 계획). 브로커 호출·
  DB 쓰기는 얇은 오케스트레이터가 담당해 결정론적 테스트가 가능하게 한다.
- **FR-008**: 워커 틱에 체결 동기화를 **라이브 모드 전용**으로 연결한다. 재평가
  cadence(기본 5초)로 1Hz 틱 비용을 제한하고, 열린 주문이 0건이면 브로커를 호출하지
  않는다. paper 모드는 호출하지 않는다(동작 무변경).
- **FR-009**: 체결 동기화 중 어떤 예외도 워커 거래 루프를 멈추지 않는다 — ERROR
  감사를 남기고 그 틱을 계속 진행한다(거래 무중단).
- **FR-010**: CLI에 `auto-invest fills [--sync]`를 추가한다. `--sync`는 한 번
  동기화하고 적용 건수를 출력, 인자 없으면 열린 주문·최근 체결을 읽기 전용 요약.

### Key Entities

- **BrokerExecution** — 브로커가 보고한 한 주문의 체결 상태(정규화): `kis_order_id`,
  `symbol`, `filled_qty`(누적), `avg_fill_price_usd`, `unfilled_qty`(옵션),
  `side`(옵션), `terminal`(종료 여부, 옵션).
- **PlannedFill / PlannedTransition** — 순수 계획 함수의 출력. 적용할 `FILL`
  (correlation_id·symbol·side·qty·price·kis_fill_id)과 상태 전이 목록.
- **FILL 감사 이벤트 / fills row** — 기존 스키마 재사용(추가 없음).
- **새 주문 상태** — `PARTIALLY_FILLED`·`FILLED`·`EXPIRED`(`orders.state` 자유 TEXT).

## Success Criteria *(mandatory)*

- **SC-001 (체결 정확성)**: 전량·부분·다단계 체결이 `fills` 합계 = 브로커 누적
  체결량과 항상 일치한다(US1·US2).
- **SC-002 (멱등)**: 임의 횟수 재동기화해도 `fills`·`current_positions`·`orders`
  상태가 1회 동기화 결과와 동일하다(SC 핵심).
- **SC-003 (브레이커 라이브 활성)**: 라이브 매도 손실 체결이 기록되면 스펙 011
  성과가 그 손실을 반영하고, 한도 초과 시 스펙 014 브레이커가 트립한다(US3).
- **SC-004 (한 잣대)**: 라이브 체결로 재구성한 보유/손익이 스펙 011 성과 엔진·정합성
  과 같은 평균단가 규약을 쓴다(헌법 원칙 X).
- **SC-005 (거래 무중단)**: 체결 동기화 오류가 발생해도 워커는 다음 틱을 정상
  진행한다(SC-001 거래 우선).
- **SC-006 (페이퍼 무영향)**: paper 모드 전체 테스트가 그대로 통과한다 — 체결
  동기화는 라이브 전용이라 paper 경로를 한 줄도 바꾸지 않는다.
- **SC-007 (Kernel 미터치)**: `git diff`에 Kernel 파일(K1~K6·K-meta) 변경 0건.

## 안전 경계 (요약)

- 주문을 내거나 취소하지 않는다 — 브로커가 확인한 체결만 기록.
- 멱등 — 재폴링 안전.
- Kernel 터치 0건 — 기존 감사 이벤트 재사용, 마이그레이션 없음.
- 거래 무중단 — 동기화 오류 격리.
- 라이브 전용 — paper 동작 불변.
- 승격/배포/캐너리/실거래 토글과 무관 — 라이브 worker는 dry-run 그대로.
