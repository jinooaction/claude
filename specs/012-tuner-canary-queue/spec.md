## 배경 (왜 이 기능인가)

스펙 005 자율 튜너는 측정 → 분석 → **행동** 루프에서 "행동" 단계를 닫는 엔진이다.
그러나 행동에는 두 갈래가 있다:

- **L1(저위험·가역)** — 즉시 자동 적용 (현재 `config/llm_kpi_thresholds.toml` 임계값 조이기 한 종류만 작동).
- **L2/L3(위험·캐너리 필요)** — 안전 검증을 거쳐야 하는 변경 (모델 라우팅 등 LLM 비용/품질에 영향).

오늘 튜너는 `cost_drift`·`latency_degradation`·`cache_miss` 드리프트를 감지하지만,
비용·지연 드리프트의 실제 수정 지점인 **모델 라우팅**(`judgment/registry.py`)은
분류기가 **L2(캐너리)** 로 강등한다(헌법 III 준수). 문제는 **L2 캐너리 경로가
"감사 로그 한 줄 적고 끝"인 빈 껍데기**(`tuner/runner.py` 의 `canary_entered`
처리)라는 것이다 — 후보가 실제로 캐너리에 들어가 검증되지 않는다. 결과적으로
위험하지만 가치 있는 자율 개선이 **갈 곳 없이 사라진다**.

스펙 007 하드닝 캐너리 패키지는 docstring에 "future spec 005 tuner … consume
from here" 라고 명시 — 튜너가 캐너리를 소비하는 것이 **설계 의도**다. 이 기능은
그 의도를 실제로 배선한다: 튜너 L2/L3 후보를 하드닝 캐너리로 **자동 투입**해
과거 리플레이 + 합성 충격 + 퍼즈로 **검증**하고 합격/불합격을 기록한다.

**핵심 안전 불변**: 캐너리 검증은 **시뮬레이션**(과거/합성 데이터)이지 실거래나
실배포가 아니다. 후보가 캐너리를 **합격해도 라이브 워커로 자동 승격하지 않는다**.
라이브 승격은 여전히 운영자/스펙 006 배포 게이트 전용(헌법 IX.B-2). 이 기능은
"위험한 후보를 안전 게이트로 검증하고 그 증거를 남기는" 단계까지만 자동화한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — L2/L3 후보가 빈 껍데기 대신 실행 가능한 캐너리 후보 기록을 남긴다 (Priority: P1)

운영자(또는 다음 세션)가 `auto-invest tune --apply` 를 돌린다. 튜너가 모델 라우팅
드리프트 같은 L2 후보를 만들면, 지금처럼 감사 로그 한 줄만 적고 버리는 대신
**캐너리가 평가할 수 있는 구조화된 후보 기록**을 남긴다: 대상 파일, 구체적
제안 변경(무엇을 무엇으로), 권장 캐너리 등급/윈도, 감지 근거. 이 기록은
튜너 리포트와 추가-전용 감사 로그 양쪽에 남아 누구나 조회할 수 있다.

**Why this priority**: 현재 L2/L3 후보는 "canary_entered" 라는 이름과 달리
어디에도 실제로 들어가지 않는 죽은 분기다. 후보를 **실행 가능한 형태로
구체화**하는 것이 다른 모든 단계의 선행 조건이다. 이것만 있어도 운영자가
수동으로 캐너리를 돌릴 수 있는 실질 가치가 생긴다(MVP).

**Independent Test**: 모델 라우팅을 L2로 분류시키는 입력으로 튜너를 apply 모드
실행 → 튜너 리포트와 감사 로그에 (대상 경로 + old→new + 권장 등급/윈도를 담은)
구조화된 캐너리 후보 기록이 정확히 한 건 생기는지 검증. dry-run 에서는 무변경.

**Acceptance Scenarios**:

1. **Given** 모델 라우팅 드리프트를 L2로 분류시키는 KPI 입력, **When** `auto-invest tune --apply` 실행, **Then** 튜너 리포트에 해당 후보가 "캐너리 후보"로 구체적 변경 내용과 함께 나타난다.
2. **Given** 같은 입력, **When** apply 모드 실행, **Then** 추가-전용 감사 로그에 캐너리 후보 기록 이벤트가 한 건(멱등 — 같은 세션 재실행 시 중복 없음) 추가된다.
3. **Given** dry-run 모드, **When** 튜너 실행, **Then** 설정·감사·git 상태를 전혀 바꾸지 않고 후보를 분석 결과로만 보고한다(재현성 보장).

### User Story 2 — 튜너가 후보를 하드닝 캐너리에 자동 투입해 검증한다 (Priority: P2)

apply 모드에서 튜너가 L2/L3 캐너리 후보를 만들면, 그 변경을 캐너리가 비교할 수
있는 형태로 구체화하고 하드닝 캐너리를 **실제로 호출**해 과거 리플레이 + 합성
충격 + 퍼즈로 검증한다. 검증이 끝나면 합격/불합격(과 불합격 시 어떤 지표가
밴드를 벗어났는지)을 추가-전용 감사 이벤트와 튜너 리포트에 기록한다.

**Why this priority**: 이것이 측정→행동 루프의 "위험-변경 검증" 팔을 실제로
닫는 핵심이다. 캐너리는 헌법 VI 단계적 확장의 안전 게이트이며, 튜너가 이를
호출함으로써 위험하지만 가치 있는 자율 개선이 안전하게 평가된다. P1(후보
구체화) 위에서 동작하므로 P2.

**Independent Test**: 과거 리플레이 데이터가 있는 환경에서 L2 후보가 나오는
입력으로 튜너 apply 실행 → 캐너리 실행이 한 번 일어나고, 그 결과(합격/불합격 +
실패 지표)가 감사 로그와 튜너 리포트에 기록되는지 검증.

**Acceptance Scenarios**:

1. **Given** L2 캐너리 후보 + 캐너리 리플레이 데이터 사용 가능, **When** 튜너 apply 실행, **Then** 하드닝 캐너리가 그 후보를 후보 rev로, 직전 합격 기준(또는 main)을 baseline rev로 한 번 평가한다.
2. **Given** 캐너리가 합격을 반환, **When** 검증 종료, **Then** 감사 로그에 "후보 캐너리 합격"이 기록되고, 튜너 리포트가 합격을 표시하되 **"라이브 미승격 — 운영자 게이트"** 를 명시한다.
3. **Given** 캐너리가 불합격을 반환, **When** 검증 종료, **Then** 감사 로그·리포트에 불합격과 밴드를 벗어난 지표 목록이 기록되고, 후보는 적용/승격되지 않는다.
4. **Given** 캐너리 리플레이 데이터가 없는 환경(과거 가격 데이터 없음), **When** 튜너 apply 실행, **Then** 캐너리는 안전하게 건너뛰고(fail-safe) 후보는 P1의 "캐너리 후보(미검증)" 기록으로만 남으며 튜너 실행 자체는 정상 종료한다.

### User Story 3 — 검증은 시뮬레이션일 뿐 어떤 자동 승격·실거래도 일으키지 않는다 (Priority: P1, 안전 게이트)

캐너리 자동 투입이 합격을 내든 불합격을 내든, 라이브 워커가 받는 코드/설정은
한 글자도 바뀌지 않고 실주문도 나가지 않는다. 후보를 검증하느라 만든 임시
작업물(임시 브랜치/커밋)은 원격으로 푸시되지 않고 검증 후 정리된다. 라이브
승격은 오직 운영자/스펙 006 배포 게이트를 통해서만 일어난다.

**Why this priority**: 이 기능의 안전 경계 그 자체다. 자동 투입이 "검증"을
넘어 "배포"로 번지면 헌법 IX.B-2(자율 배포는 하드닝 캐너리가 유일 경로이며
그조차 운영자 게이트)·VIII.A(장중 배포 금지)를 침해한다. P1으로 둬서 절대
타협하지 않는다.

**Independent Test**: 튜너 apply 실행 전후로 (1) `judgment/registry.py` 등 후보
대상 파일의 작업트리 내용, (2) 라이브 워커 설정, (3) 원격 브랜치 목록이
불변임을 검증. 캐너리 합격 후보가 있어도 배포·승격 이벤트가 감사 로그에 전혀
없음을 검증.

**Acceptance Scenarios**:

1. **Given** 캐너리를 합격한 L2 후보, **When** 튜너 실행 종료, **Then** 후보 대상 파일의 작업트리 내용이 실행 전과 동일하다(검증용 임시 변경은 정리됨).
2. **Given** 어떤 캐너리 결과든, **When** 튜너 실행 종료, **Then** 감사 로그에 라이브 배포·승격 이벤트가 추가되지 않는다(검증 이벤트만 추가).
3. **Given** 검증을 위해 임시 브랜치/커밋이 필요한 경우, **When** 튜너가 그것을 만들면, **Then** 원격(origin)으로 푸시하지 않고 검증 후 삭제한다.

### Edge Cases

- **Kernel 터치 후보**: 분류기가 무조건 L4로 강등 → 캐너리 자동 투입 대상에서 제외(L4는 인간 머지 경로). 캐너리에 절대 자동 투입되지 않는다.
- **장 시간**: L1 적용과 마찬가지로 캐너리 투입도 측정 게이트(헌법 X)를 존중한다. 캐너리 검증은 시뮬레이션이라 장중 배포(VIII.A)에 해당하지 않지만, 측정 표본 부족이면 후보 자체가 안 만들어진다.
- **멱등성**: 같은 세션 날짜에 같은 후보를 재실행하면 캐너리를 중복 투입하지 않는다(이미 검증한 후보 dedup).
- **캐너리 내부 오류**(EXIT_INTERNAL): 튜너는 후보를 "검증 실패(내부 오류)"로 기록하고 다른 후보 처리를 계속한다 — 한 후보의 캐너리 오류가 전체 튜너 실행을 중단시키지 않는다.
- **여러 L2/L3 후보**: 각 후보가 독립적으로 검증된다. 한 후보 불합격이 다른 후보 검증을 막지 않는다.
- **baseline 부재**: 직전 합격 캐너리 기록이 없으면 `origin/main`(또는 현재 HEAD)을 baseline으로 사용한다(기존 `run_canary` 의 baseline 해석 재사용).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-C12-01**: 시스템은 튜너 분류 결과가 L2 또는 L3인 후보에 대해, 캐너리가 평가할 수 있는 **구조화된 캐너리 후보 기록**(대상 파일 경로, 구체적 제안 변경(old→new), 권장 캐너리 등급, 감지 근거, 측정 표본 수)을 생성해야 한다.
- **FR-C12-02**: 시스템은 캐너리 후보 기록을 **추가-전용 감사 로그**(헌법 IV)에 기록해야 하며, 같은 세션·같은 후보 재실행 시 중복 기록하지 않아야 한다(멱등).
- **FR-C12-03**: 시스템은 캐너리 후보를 **튜너 리포트**에 표시하되, 검증 전/후 상태(미검증·합격·불합격·검증 건너뜀)를 구분해 보여야 한다.
- **FR-C12-04**: apply 모드에서 시스템은 캐너리 리플레이 데이터가 사용 가능하면 각 L2/L3 후보를 **하드닝 캐너리(`run_canary`)에 한 번 투입**해 과거 리플레이 + 합성 충격 + 퍼즈로 검증해야 한다.
- **FR-C12-05**: 시스템은 캐너리 후보를 캐너리가 비교할 수 있는 후보 rev로 **구체화**해야 하며, 이를 위해 만든 임시 git 작업물은 **원격으로 푸시하지 않고** 검증 후 정리해야 한다.
- **FR-C12-06**: 시스템은 각 캐너리 검증의 결과(합격/불합격, 불합격 시 밴드를 벗어난 지표 목록, 캐너리 run id)를 **추가-전용 감사 이벤트와 튜너 리포트**에 기록해야 한다.
- **FR-C12-07**: 시스템은 캐너리 합격 후보를 **라이브 워커로 자동 승격·배포해서는 안 된다**. 합격은 "검증됨, 운영자/스펙 006 게이트 대기" 상태로만 기록한다(헌법 IX.B-2).
- **FR-C12-08**: 시스템은 Kernel(K1~K6·K-meta) 터치 후보(분류 L4)를 **캐너리 자동 투입 대상에서 제외**해야 한다(기존 L4 인간 머지 경로 유지).
- **FR-C12-09**: 시스템은 캐너리 리플레이 데이터가 없는 환경에서 캐너리 투입을 **안전하게 건너뛰고**(fail-safe) 후보를 "미검증 캐너리 후보"로만 기록하며, 튜너 실행 자체는 성공(종료 0)으로 끝내야 한다.
- **FR-C12-10**: 시스템은 한 후보의 캐너리 내부 오류가 **다른 후보 검증이나 전체 튜너 실행을 중단시키지 않도록** 격리해야 한다(해당 후보만 오류로 기록).
- **FR-C12-11**: dry-run 모드에서 시스템은 캐너리를 투입하지 않고 설정·감사·git 상태를 전혀 바꾸지 않으며, 후보를 분석 결과로만 보고해야 한다(재현성·무변경).
- **FR-C12-12**: 캐너리 후보 구체화·검증·기록은 **결정론적**이어야 한다 — 같은 입력(KPI 스냅샷 + 매니페스트 + 리플레이 윈도)이면 같은 후보·같은 캐너리 입력을 만든다(LLM 미호출).

### Key Entities

- **캐너리 후보 기록(CanaryCandidate)**: L2/L3 분류 후보를 캐너리가 평가할 수 있게 구체화한 단위. 속성: 후보 id, 감지 규칙, 대상 파일 경로, 구체적 변경(old→new), 권장 캐너리 등급(tier), 권장 윈도, 측정 표본 수, 근거.
- **캐너리 검증 결과(CanaryValidationResult)**: 한 후보를 캐너리에 투입한 결과. 속성: 후보 id, 캐너리 run id, 결과(passed/failed/skipped/internal_error), 불합격 지표 목록, baseline/candidate rev.
- **감사 이벤트**: 추가-전용 `AUTO_TUNED_*` 계열의 캐너리 후보·검증 이벤트(K4 추가-전용 터치, 기존 패턴 따름).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-C12-01**: L2/L3 후보가 더 이상 "감사 로그 한 줄"로 사라지지 않는다 — apply 모드에서 100%의 L2/L3 후보가 (검증됨/미검증/건너뜀) 구조화된 캐너리 후보 기록으로 남는다.
- **SC-C12-02**: 캐너리 리플레이 데이터가 있는 환경에서, L2/L3 후보의 캐너리 검증 결과(합격/불합격 + 실패 지표)가 감사 로그와 튜너 리포트에서 운영자가 별도 도구 없이 조회 가능하다.
- **SC-C12-03**: 캐너리 검증 전후로 후보 대상 파일의 작업트리·라이브 워커 설정·원격 브랜치가 0건 변경된다(안전 불변 측정).
- **SC-C12-04**: 캐너리 합격 후보가 있어도 라이브 배포·승격 이벤트가 감사 로그에 0건 추가된다(자동 승격 금지 측정).
- **SC-C12-05**: 같은 입력으로 dry-run을 두 번 돌리면 동일한 후보 집합을 보고하고(결정성), 설정·감사·git 상태 변경은 0건이다.
- **SC-C12-06**: 캐너리 리플레이 데이터가 없거나 한 후보가 캐너리 내부 오류를 내도 튜너 실행은 종료 0으로 끝나고 나머지 후보는 정상 처리된다(견고성).
- **SC-C12-07**: 새 테스트가 전부 통과하고 기존 전체 테스트 스위트(현재 902 통과)에 회귀가 없다.

## Assumptions

- **하드닝 캐너리(`run_canary`)는 안정적 공개 인터페이스**다(스펙 007 출시 완료). 캐너리는 후보 rev vs baseline rev의 git 트리를 비교하고 리플레이/충격/퍼즈로 평가해 합격/불합격을 반환한다. 이 기능은 캐너리 내부를 바꾸지 않고 **소비만** 한다.
- 캐너리는 후보를 **git rev**로 받는다. 따라서 튜너는 L2/L3 변경을 임시 git 커밋으로 구체화해 후보 rev를 만든다(로컬·임시·미푸시·검증 후 삭제).
- 캐너리 검증에는 **과거 리플레이 데이터**(스펙 008 백테스트 데이터)가 필요하다. 데이터가 없는 환경에서는 검증을 건너뛰고 후보만 기록한다(FR-C12-09 fail-safe).
- 모델 라우팅(`judgment/registry.py`)은 비커널 `/judgment/` 경로라 기존 분류기가 이미 L2로 강등한다 — 이 기능은 분류 규칙을 바꾸지 않는다.
- 캐너리 후보 감지(어떤 드리프트가 어떤 구체적 모델 라우팅 변경을 제안하는지)는 결정론적 규칙으로 구현한다 — LLM 미호출. 구체적 노브 매핑(예: cost_drift → 더 저렴한 모델로 라우팅)은 plan 단계에서 확정한다.
- 이 기능은 작업 브랜치 `claude/upbeat-bell-Sq9qm` 위에서 진행하고 새 브랜치를 만들지 않는다.

## 안전 경계 (헌법 준수 — 절대 협상 불가)

- **헌법 IX.B-2**: 하드닝 캐너리는 자율 배포의 유일 경로이며 그조차 운영자 게이트. 이 기능은 캐너리를 **검증용**으로만 호출 — 배포·승격은 일으키지 않는다.
- **헌법 IV(추가-전용 감사)**: 모든 후보·검증 기록은 추가-전용. K4 터치는 기존 `AUTO_TUNED_*` 패턴을 따르는 추가-전용(이벤트 타입 추가)이며, 그 외 Kernel(K1·K2·K3·K5·K6·K-meta) 터치 0건을 목표로 한다.
- **헌법 VIII.A(장중 배포 금지)**: 캐너리 검증은 시뮬레이션이라 배포가 아니다. 실제 라이브 승격은 이 기능 범위 밖(운영자/스펙 006).
- **헌법 X(측정 기반)**: 측정 표본 부족이면 후보 자체가 안 만들어진다(기존 게이트 재사용).
- **Kernel L4 강등 불변**: Kernel 터치 후보는 캐너리 자동 투입 대상이 아니다.
