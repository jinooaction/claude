# HANDOFF 021 — 장 마감 정합성 자동 실행 (스펙 001 T050/T052 완성, 2026-05-27)

PR #75 머지 커밋 `4319535`. **로컬 장부와 브로커 보유를 매 장 마감마다 자동으로
대조해 드리프트를 잡는** 정합성 검증의 자동 호출 배선을 채웠습니다. 운영자 지시
("세계 최고 수준 작업 분석·우선순위·자율 수행")의 코드 분석 1순위로 선택했습니다.

## 왜 1순위였나 (조용한 상태 드리프트 = 자율 운영의 가장 큰 안전 구멍)

스펙 001 P2 사용자 스토리 "조용한 상태 드리프트 방지"의 키스톤인 정합성 검증은
**구현(T049)·테스트(T048)는 됐는데 자동 호출하는 배선(T050)이 통째로 빠져
있었습니다.** 워커 틱·`run_forever`·systemd 타이머 어디에도 호출이 없고 `reconcile`
CLI 명령도 없어서, **유일한 호출자가 테스트 스위트**였습니다. 그 결과:

- 라이브(dry-run) 자율 운영 중 로컬 장부와 브로커 보유 간 불일치를 **한 번도 잡지
  못했음** — 잘못된 보유 그림 위에서 거래할 위험.
- 스펙 013 헬스 롤업의 정합성 점검이 기록이 없어 **영구 DEGRADED**.
- `reconcile_now` docstring 이 "CLI `reconcile` 서브커맨드 / 스케줄 세션-마감
  경로가 이 진입점을 쓴다"고 적혀 있었으나 **둘 다 없던 거짓 주장**.
- 직전 스펙 015(라이브 체결 동기화)가 체결을 장부에 기록하게 만든 지금, 이 검증이
  비로소 실제 신호를 가짐.

## 한 줄 요약

- **`worker/loop.py`** — 세션이 열림→닫힘으로 바뀌는 첫 틱에 정합성 1회 자동 실행
  (`Worker._session_was_open` 추적 + `_reconcile_at_close()`). 한 닫힘 구간에 정확히
  1회(전이 직후 플래그 False → 같은 구간 이후 틱은 미트리거).
- **`cli.py`** — `auto-invest reconcile` 명령 추가(수동/모니터링용). 같은
  `run_reconciliation` 진입점 재사용, 토큰 캐시 재사용. 종료 코드 0(OK)/1(불일치·
  결론 불가)/2(설정·DB 오류). `reconcile_now` docstring 정정.
- **테스트 7건** — 장마감 OK / 불일치 halt / paper 미실행 / 한 구간 1회만 / 시작 시
  닫힘이면 미트리거 + CLI 사용오류 2건.

## 안전 경계 (핵심)

## 설계 노트 — APScheduler 대신 틱 루프 전이 감지

스펙 001 T050 원문은 "APScheduler 로 세션 마감에 호출"이었으나, 실제 워커는
APScheduler 가 아니라 asyncio 틱 루프라서 **세션 열림→닫힘 전이 감지**로 구현했습니다.
이는 스펙 014(브레이커 인-틱)·스펙 015(체결 동기화 인-틱)와 일관되며, 둘 다 이미
틱 안에서 브로커/상태 결합 주기 작업을 합니다. 정합성도 브로커 결합 작업이라 워커가
이미 쥔 인증 브로커 클라이언트·토큰을 재사용하는 인-틱이 별도 프로세스 재인증보다
견고합니다.
